# API商业化平台计费规则改造方案

***

## 一、现状问题分析

### 1.1 当前支持的计费规则

| 规则       | 状态   |
| -------- | ---- |
| 按次计费     | ✅ 支持 |
| 阶梯计费（部分） | ✅ 支持 |
| 四网计费（部分） | ✅ 支持 |
| 套餐包计费    | ✅ 支持 |

### 1.2 当前不支持的计费规则

| 规则           | 状态    | 影响                          |
| ------------ | ----- | --------------------------- |
| 多接口次数赠送（规则5） | ❌ 不支持 | 无法实现"主接口调用N次，赠送接口免费N次"的业务场景 |
| 多接口合计阶梯（规则6） | ❌ 不支持 | 无法实现"ABC三接口合计次数，按阶梯计费"      |
| 多接口最大计费（规则7） | ❌ 不支持 | 无法实现"ABC三接口取最大值计费"          |
| 计费键计费（规则8）   | ❌ 不支持 | 无法实现"按车牌/企业代码计费"            |
| 手工记录（规则9）    | ❌ 不支持 | 无法支持特殊场景人工录单                |

### 1.3 问题根因

**当前系统设计把"单接口按次计费"当成唯一场景，表结构缺少抽象：**

| 问题              | 表现                                           |
| --------------- | -------------------------------------------- |
| 调用明细表缺少通用计量字段   | 没有 `billing_key`、`operator` 字段，无法支持计费键、运营商维度 |
| 计费规则表缺少"计费维度"抽象 | 每种计费方式都是独立的字段或代码，新增规则 = 改表或改代码               |
| 缺少接口分组概念        | 多接口组合计费需要预定义接口组，但表设计里没有                      |
| 计费配置结构单一        | 无法支持阶梯（有序列表）、分支（key-value映射）、嵌套对象等多种参数结构     |

***

## 二、目标架构

### 2.1 四层架构

```
┌─────────────────────────────────────────────────────────────┐
│  第一层：调用明细层                                          │
│  记录每次API调用原始数据                                      │
│  （不管怎么计费，先把数据完整记下来）                          │
├─────────────────────────────────────────────────────────────┤
│  第二层：规则模板层                                          │
│  定义"怎么算"                                                │
│  计费维度 + 计费键类型 + 组合策略 + 定价模型 + 参数配置        │
├─────────────────────────────────────────────────────────────┤
│  第三层：接口组层（仅多接口组合计费时使用）                      │
│  定义哪些API是一组、各自角色（主接口/赠送接口）                  │
├─────────────────────────────────────────────────────────────┤
│  第四层：客户定价层                                          │
│  具体客户 + 具体API/接口组 + 引用模板 + 填充价格                │
├─────────────────────────────────────────────────────────────┤
│  第五层：计算引擎层                                          │
│  调用明细 + 规则配置 + 客户定价 → 账单                         │
└─────────────────────────────────────────────────────────────┘
```

***

### 2.2 规则模板五要素

规则模板由以下五个要素组成：

```
规则模板 = 计费维度 + 计费键类型 + 组合策略 + 定价模型 + 参数配置
              ↓           ↓           ↓           ↓           ↓
           谁来计量    键的类型     怎么组合     怎么算      具体价格
```

#### 要素一：计费维度 —— "谁来计量"

| 计费维度         | 说明      | 支持的规则     |
| ------------ | ------- | --------- |
| SINGLE\_API  | 单个API计费 | 规则1、2、3、4 |
| API\_GROUP   | 多接口组合计费 | 规则5、6、7   |
| BILLING\_KEY | 计费键计费   | 规则8       |
| OPERATOR     | 运营商维度计费 | 规则3       |
| MANUAL       | 手工记录    | 规则9       |

#### 要素二：计费键类型 —— "键是什么"（仅BILLING\_KEY时）

| 计费键类型      | 说明            |
| ---------- | ------------- |
| CAR        | 车牌号           |
| ENTERPRISE | 企业代码/统一社会信用代码 |
| LICENSE    | 营业执照号         |
| DEVICE     | 设备ID          |
| CUSTOM     | 自定义           |

#### 要素三：组合策略 —— "多接口怎么组合"（仅API\_GROUP时）

| 组合策略           | 说明       | 对应规则            |
| -------------- | -------- | --------------- |
| SUM            | 合计次数     | 规则6：ABC三接口次数加起来 |
| MAX            | 取最大次数    | 规则7：ABC三接口取最大   |
| PRIMARY\_BONUS | 主接口+赠送接口 | 规则5             |

#### 要素四：定价模型 —— "怎么算"

| 模型      | 说明             |
| ------- | -------------- |
| FIXED   | 固定单价           |
| TIERED  | 阶梯计价           |
| BRANCH  | 分支定价（按条件查表）    |
| CAPPED  | 带上限（固定单价+上限控制） |
| PACKAGE | 套餐包            |

#### 要素五：参数配置 —— "具体填什么价格"

参数配置需要支持多种数据结构：

| 数据结构        | 说明                                          | 适用场景   |
| ----------- | ------------------------------------------- | ------ |
| 单值          | `{"PRICE": 0.1}`                            | 固定单价   |
| 有序列表        | `{"tiers": [{max: 1000, price: 0.1}, ...]}` | 阶梯计价   |
| key-value映射 | `{"branch": {"移动": 0.1, "联通": 0.2}}`        | 四网计费   |
| 嵌套对象        | `{primary: {...}, bonus: [...]}`            | 主+赠送接口 |

***

### 2.3 各规则的五要素对照

| 规则   | 计费维度         | 计费键类型          | 组合策略           | 定价模型           | 参数结构        |
| ---- | ------------ | -------------- | -------------- | -------------- | ----------- |
| 1.按次 | SINGLE\_API  | -              | -              | FIXED          | 单值          |
| 2.阶梯 | SINGLE\_API  | -              | -              | TIERED         | 有序列表        |
| 3.四网 | SINGLE\_API  | -              | -              | BRANCH         | key-value映射 |
| 4.套餐 | SINGLE\_API  | -              | -              | PACKAGE        | 单值+额度       |
| 5.赠送 | API\_GROUP   | -              | PRIMARY\_BONUS | PRIMARY\_BONUS | 嵌套对象        |
| 6.合计 | API\_GROUP   | -              | SUM            | TIERED         | 有序列表        |
| 7.最大 | API\_GROUP   | -              | MAX            | FIXED          | 单值          |
| 8.键计 | BILLING\_KEY | CAR/ENTERPRISE | -              | CAPPED         | 单值+上限       |
| 9.手工 | MANUAL       | -              | -              | -              | -           |

## 三、表结构设计

### 3.1 调用明细表（改造）

**现状问题**：缺少扩展字段，无法支持计费键、运营商维度。

**改造方案**：新增扩展字段，原有字段保留。

```sql
ALTER TABLE 调用明细表
ADD COLUMN 计费键 VARCHAR(100) COMMENT '计费键（车牌号/企业代码等）',
ADD COLUMN 计费键类型 VARCHAR(20) COMMENT '计费键类型：CAR/ENTERPRISE/LICENSE/DEVICE/CUSTOM',
ADD COLUMN 运营商 VARCHAR(20) COMMENT '运营商：移动/联通/电信/广电',
ADD COLUMN 原始数据 JSON COMMENT '原始数据JSON，存储上游所有字段';
```

**说明**：

- `计费键` 和 `计费键类型` 为规则8（计费键计费）提供数据基础
- `运营商` 为规则3（四网计费）提供精确数据
- `原始数据` 为未来可能的扩展预留，不必每次改表

***

### 3.2 规则模板表（新建）

**作用**：定义计费规则的"骨架"，包含五个要素。

```sql
CREATE TABLE 规则模板表 (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    模板编码 VARCHAR(50) NOT NULL UNIQUE COMMENT '如：PRIMARY_BONUS、SUM_TIERED',
    模板名称 VARCHAR(100) NOT NULL COMMENT '如：多接口次数赠送',

    -- 要素一：计费维度
    计费维度 VARCHAR(20) NOT NULL COMMENT 'SINGLE_API / API_GROUP / BILLING_KEY / OPERATOR / MANUAL',

    -- 要素二：计费键类型（仅BILLING_KEY时使用）
    计费键类型 VARCHAR(20) COMMENT 'CAR / ENTERPRISE / LICENSE / DEVICE / CUSTOM',

    -- 要素三：组合策略（仅API_GROUP时使用）
    组合策略 VARCHAR(20) COMMENT 'SUM / MAX / PRIMARY_BONUS',

    -- 要素四：定价模型
    定价模型 VARCHAR(20) NOT NULL COMMENT 'FIXED / TIERED / BRANCH / CAPPED / PACKAGE / PRIMARY_BONUS',

    -- 要素五：参数定义（JSON，定义参数结构和类型）
    参数定义 JSON NOT NULL COMMENT '参数变量定义，支持多种数据结构',

    -- 计算公式描述
    计算公式 VARCHAR(500) COMMENT '计算公式描述，用于文档和预览',

    状态 TINYINT DEFAULT 1 COMMENT '1:启用 0:停用',
    创建时间 DATETIME DEFAULT CURRENT_TIMESTAMP,
    更新时间 DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_维度 (计费维度),
    INDEX idx_状态 (状态)
) COMMENT '计费规则模板表';
```

**参数定义JSON结构说明**：

```json
// 单值类型（按次计费）
{
    "PRICE": {
        "type": "DECIMAL",
        "label": "单价",
        "unit": "元/次",
        "required": true
    }
}

// 有序列表类型（阶梯计价）
{
    "tiers": {
        "type": "TIER_LIST",
        "label": "阶梯配置",
        "items": {
            "max": {"type": "INTEGER", "label": "上限"},
            "price": {"type": "DECIMAL", "label": "单价"}
        }
    }
}

// key-value映射类型（四网计费）
{
    "branch_config": {
        "type": "KEY_VALUE",
        "label": "各运营商单价",
        "items": {
            "移动": {"price": {"type": "DECIMAL"}},
            "联通": {"price": {"type": "DECIMAL"}},
            "电信": {"price": {"type": "DECIMAL"}},
            "广电": {"price": {"type": "DECIMAL"}}
        }
    }
}

// 嵌套对象类型（主+赠送）
{
    "primary": {
        "type": "OBJECT",
        "label": "主接口配置",
        "properties": {
            "api_id": {"type": "STRING", "label": "接口"},
            "price": {"type": "DECIMAL", "label": "单价"}
        }
    },
    "bonus": {
        "type": "LIST",
        "label": "赠送接口列表",
        "items": {
            "type": "OBJECT",
            "properties": {
                "api_id": {"type": "STRING", "label": "接口"},
                "overage_price": {"type": "DECIMAL", "label": "超量单价"}
            }
        }
    }
}
```

**预置数据**：

| 模板编码                     | 模板名称    | 计费维度         | 计费键类型      | 组合策略           | 定价模型           |
| ------------------------ | ------- | ------------ | ---------- | -------------- | -------------- |
| FIXED                    | 标准按次计费  | SINGLE\_API  | -          | -              | FIXED          |
| TIERED                   | 单接口阶梯计费 | SINGLE\_API  | -          | -              | TIERED         |
| BRANCH                   | 四网计费    | SINGLE\_API  | -          | -              | BRANCH         |
| PACKAGE                  | 套餐包     | SINGLE\_API  | -          | -              | PACKAGE        |
| PRIMARY\_BONUS           | 多接口次数赠送 | API\_GROUP   | -          | PRIMARY\_BONUS | PRIMARY\_BONUS |
| SUM\_TIERED              | 多接口合计阶梯 | API\_GROUP   | -          | SUM            | TIERED         |
| GROUP\_MAX               | 多接口最大计费 | API\_GROUP   | -          | MAX            | FIXED          |
| BILLING\_KEY\_CAPPED     | 计费键计费   | BILLING\_KEY | CAR        | -              | CAPPED         |
| BILLING\_KEY\_ENTERPRISE | 企业代码计费  | BILLING\_KEY | ENTERPRISE | -              | CAPPED         |

### 3.3 接口组表（新建）

**作用**：定义API的组合关系，支持多接口组合计费（规则5、6、7）。

```sql
-- 接口组主表
CREATE TABLE 接口组表 (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    接口组编码 VARCHAR(50) NOT NULL UNIQUE COMMENT '如：CAR_SERVICE_GROUP',
    接口组名称 VARCHAR(100) NOT NULL COMMENT '如：车辆服务组合',
    描述 VARCHAR(255),
    状态 TINYINT DEFAULT 1,
    创建时间 DATETIME DEFAULT CURRENT_TIMESTAMP,
    更新时间 DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_编码 (接口组编码)
) COMMENT '接口组表';

-- 接口组成员表
CREATE TABLE 接口组成员表 (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    接口组id BIGINT NOT NULL,
    api_id VARCHAR(50) NOT NULL,
    角色 VARCHAR(20) NOT NULL COMMENT 'PRIMARY:主接口 BONUS:赠送接口',
    排序 INT DEFAULT 0 COMMENT '用于展示顺序',
    创建时间 DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (接口组id) REFERENCES 接口组表(id),
    INDEX idx_接口组id (接口组id),
    INDEX idx_api (api_id)
) COMMENT '接口组成员表';
```

**示例数据**：

接口组表：

| id | 接口组编码            | 接口组名称  |
| -- | ---------------- | ------ |
| 1  | CAR\_SERVICE     | 车辆服务组合 |
| 2  | ENTERPRISE\_INFO | 企业信息组合 |

接口组成员表：

| id | 接口组id | api\_id | 角色      |
| -- | ----- | ------- | ------- |
| 1  | 1     | API-001 | PRIMARY |
| 2  | 1     | API-002 | BONUS   |
| 3  | 1     | API-003 | BONUS   |
| 4  | 2     | API-101 | PRIMARY |
| 5  | 2     | API-102 | BONUS   |

### 3.4 客户定价表（改造）

**现状问题**：只支持绑定单个API，无法绑定接口组。

**改造方案**：支持绑定单个API或接口组，引用规则模板，填充价格参数。

```sql
ALTER TABLE 客户定价表
ADD COLUMN target_type VARCHAR(20) NOT NULL DEFAULT 'SINGLE_API' COMMENT 'SINGLE_API:单个API API_GROUP:接口组',
ADD COLUMN target_id BIGINT COMMENT 'api_id或接口组id，根据target_type区分',
ADD COLUMN rule_template_id BIGINT COMMENT '引用的规则模板id',
ADD COLUMN 填充参数 JSON COMMENT '模板参数值，根据模板的参数定义填充';
```

**说明**：

- `target_type` 区分是单个API还是接口组
- `target_id` 根据target\_type存储api\_id或接口组id
- `rule_template_id` 引用规则模板表
- `填充参数` 是模板参数的具体值，格式根据参数定义的type决定

**填充参数示例**：

```json
// 按次计费
{"PRICE": 0.08}

// 阶梯计费
{"tiers": [{"max": 1000, "price": 0.1}, {"max": null, "price": 0.05}]}

// 四网计费
{"branch_config": {"移动": {"price": 0.1}, "联通": {"price": 0.2}, "电信": {"price": 0.15}, "广电": {"price": 0.15}}}

// 主+赠送计费
{
    "primary": {"api_id": "API-001", "price": 0.1},
    "bonus": [
        {"api_id": "API-002", "overage_price": 0.2},
        {"api_id": "API-003", "overage_price": 0.15}
    ]
}

// 计费键计费（带上限）
{"PRICE": 0.1, "daily_cap": 50, "monthly_cap": 800}
```

## 四、改造优先级与步骤

### 4.1 优先级

| 优先级    | 改造内容           | 影响范围          | 理由    |
| ------ | -------------- | ------------- | ----- |
| **P0** | 接口组表 + 客户定价表扩展 | 直接支持规则5、6、7   | 业务急需  |
| **P1** | 规则模板表          | 为新规则提供模板支撑    | 依赖P0  |
| **P2** | 调用明细表扩展        | 为规则8提供数据基础    | 长期需要  |
| **P3** | 计算引擎优化         | 长期目标，减少新规则开发量 | 可逐步迭代 |

### 4.2 改造步骤

1. Phase 1：接口组支持（P0）
2. **目标**：支持多接口赠送、合计阶梯、最大计费

**改造内容**：

1. 新建接口组表、接口组成员表
2. 改造客户定价表，支持target\_type和target\_id
3. 开发接口组管理功能（CRUD）
4. 改造客户定价页面，支持选择接口组

**影响**：

- 现有计费逻辑不受影响
- 新规则使用新表

***

#### Phase 2：规则模板支持（P1）

**目标**：提供规则模板支撑，支持灵活配置

**改造内容**：

1. 新建规则模板表
2. 预置常用模板数据
3. 改造计算引擎，支持按模板计算（根据五要素和参数配置）
4. 开发规则模板管理页面

**影响**：

- 现有规则可逐步迁移到模板
- 新规则直接使用模板

***

#### Phase 3：调用明细扩展（P2）

**目标**：支撑计费键计费

**改造内容**：

1. 调用明细表新增计费键、计费键类型、运营商字段
2. 上游数据接入改造，确保传入计费键和运营商数据
3. 开发计费键配置功能

**影响**：

- 需要上游配合传入数据
- 现有逻辑不受影响

***

#### Phase 4：计算引擎优化（P3）

**目标**：减少新规则开发量

**改造内容**：

1. 引入公式引擎，将计算逻辑配置化
2. 开发可视化配置器
3. 逐步迁移现有规则到公式配置

**说明**：

- 这是长期目标
- 可根据业务发展情况决定是否实施

***

## 五、对现有业务的影响

### 5.1 改造原则

**不破坏现有业务**，新功能使用新表，老功能逐步迁移：

| 原则   | 说明           |
| ---- | ------------ |
| 兼容现有 | 现有计费逻辑保持不变   |
| 逐步迁移 | 老规则可逐步迁移到新架构 |
| 可回滚  | 每个阶段可单独回滚    |

### 5.2 各阶段影响

| 阶段      | 对现有业务影响     | 风险            |
| ------- | ----------- | ------------- |
| Phase 1 | 无影响，新增接口组功能 | 低             |
| Phase 2 | 无影响，新增模板功能  | 低             |
| Phase 3 | 无影响，新增字段可选  | 中（需上游配合）      |
| Phase 4 | 无影响，引擎升级    | 中（公式引擎需要充分测试） |

### 5.3 回滚方案

每个阶段独立部署，发现问题可单独回滚：

- Phase 1 回滚：删除接口组表，恢复客户定价表
- Phase 2 回滚：删除规则模板表相关逻辑
- Phase 3 回滚：删除新增字段（需注意数据迁移）

| 调整现有规则价格 | 需要改数据库  | 运营直接改配置        |
| 新增计费维度   | 需要开发改代码 | 加模板数据          |
| 多接口组合计费  | 需要开发改代码 | 运营配置接口组        |
| 新增计费键类型  | 需要改表结构  | 加模板数据          |
| 新增运营商定价  | 需要改代码   | 加key-value映射配置 |
