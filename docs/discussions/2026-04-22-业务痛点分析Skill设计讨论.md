# 业务痛点分析Skill设计讨论

日期：2026-04-22

---

## 讨论背景

完成了"业务现状梳理"Skill后，需要设计下一个Skill来完成需求分析链路的下一步。

## 讨论结论

### 需求分析完整链路

```
业务现状梳理（已完成）→ 业务痛点分析（本次）→ 需求拆解 → 产品设计
```

### 痛点分析和现状梳理的区别

| 维度 | 业务现状梳理 | 业务痛点分析 |
|------|-------------|-------------|
| 目的 | 描述客观事实 | 发现改进机会 |
| 产出 | 角色、流程、系统、数据 | 痛点清单、优先级、改进价值 |
| 访谈重点 | "做了什么、怎么做的" | "哪里不顺、为什么不顺、影响多大" |
| 结论性质 | 中立描述 | 带有判断和排序 |

### 痛点分析数据来源

1. 从现状梳理报告中提取线索（可能遗漏）
2. 通过补充访谈深挖线索和盲区
3. 综合输出痛点分析

### 整合的方法论

| 阶段 | 融入方法论 |
|------|-----------|
| 痛点识别 | JTBD（Jobs-to-be-Done）+ 用户旅程图 |
| 痛点验证 | 访谈追问 + 量化确认 |
| 根因追溯 | 5 Why + 鱼骨图（人机料法环测）|
| 痛点定级 | 影响范围/频率 + Kano模型 |
| 优先级排序 | Impact vs Effort + 业务战略对齐 |

### 基础原则更新

增加了"分析师自主判断"原则：痛点优先级由分析师根据线索可信度、影响范围、商业价值自主判断，无需询问用户。判断后直接说明理由，让用户确认。

### 待实验

Skill框架搭好后，需要找实际业务场景验证效果。本次准备用API商业化业务现状梳理报告作为输入进行实验，但暂时搁置，待后续继续。
