# 调研Skill设计与创建讨论

日期：2026-04-22

---

## 讨论背景

用户认为原有的"业务现状梳理"Skill采用对话方式梳理业务，时间较长。用户希望设计两个前置Skill，让助理先去调研收集信息，再输入给业务现状梳理Skill，提高效率。

---

## 讨论过程

### 1. 核心需求确认

用户想要的模式：
- **AI产品经理**（我）：主导设计调研计划、访谈问题
- **助理**（用户）：执行访谈、记录回答
- **业务现状梳理Skill**：整合结构化访谈记录 + 对话分析

### 2. Skill设计讨论

**两个独立Skill vs 一个整合Skill**

| 方案 | 结论 |
|------|------|
| 两个独立Skill | ✅ 更灵活，职责单一 |
| 一个整合Skill | ❌ 捆绑使用，不够灵活 |

最终决定：**两个独立Skill**

**Skill分工**：
- Skill A（调研规划）：分析业务域，确定需要调研的角色和信息类型
- Skill B（访谈问题生成）：基于调研计划生成访谈问题清单

### 3. 方法论引入

用户强调需要引入成熟方法论，而非仅基于项目现有文档。

**搜索到的成熟方法论**：

| 方法论 | 来源 |
|--------|------|
| 深度访谈 (IDI) | 用户研究 |
| 5W1H分析法 | 业务分析 |
| STAR/BEI行为事件访谈 | 人才评估/用户研究 |
| 5Why追问法 | 问题分析 |
| 角色分层（战略/战术/执行） | TOGAF/BABOK |

### 4. 双模式叠加讨论

**初始理解**：对话模式 vs 访谈记录模式 = 二选一

**修正后的理解**：
- 访谈记录 = 客观信息（来自业务角色）
- 对话输入 = 主观分析（助理的思考和判断）
- **两者互补，不是互斥**

**最终设计**：业务现状梳理Skill支持双模式叠加整合

### 5. 判断机制

**方案A（内容识别）**：自动识别结构化字段
- 优点：用户无感知
- 缺点：可能误判

**方案B（显式标记）**：用户添加`/interview`标记
- 优点：简单准确
- 缺点：增加负担

**最终决定**：方案A（内容识别）为主，降级处理

## 创建的Skill

### 1. 调研规划
- 路径：`.trae/skills/调研规划/SKILL.md`
- 方法论：5W1H分析、角色分层模型
- 输出：调研计划书

### 2. 访谈问题生成
- 路径：`.trae/skills/访谈问题生成/SKILL.md`
- 方法论：BEI、STAR、5Why、深度访谈技巧
- 输出：每个角色的访谈问题清单 + 记录模板

### 3. 业务现状梳理（适配）
- 修改内容：支持双模式叠加（结构化访谈记录 + 对话输入）
- 新增信息来源标记：【访谈-角色A】【助理分析】【待验证】
