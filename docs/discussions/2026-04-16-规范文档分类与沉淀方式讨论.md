# 规范文档分类与沉淀方式的讨论

## 起因

在一次任务中，我意识到每次开启新任务时，不知道在其他任务中讨论过的上下文。为了让项目正常推进，需要记录上下文。但当时文档分散在业务文档、开发规范、规则文档中，没有明确的分类标准。

---

## 一、文档分类的讨论

### 初始理解

我最初理解的分类是：

| 类型 | 约束内容 |
|------|----------|
| **约束性规则** | 禁止省略分页功能、菜单箭头只用 `>` |
| **代码实现标准** | 列表页模板、分页逻辑 |

但用户指出这个理解有问题。

### 重新理解

用户提出了更清晰的分类标准：

| 类型 | 判断标准 | 示例 |
|------|----------|------|
| **约束性规则** | 与代码/业务无关，约束 AI 工作方式 | "任务结束上传 GitHub"、"有歧义先澄清" |
| **前端组件规范** | 组件选择/组合，改了会影响功能 | 列表页必须有搜索区+列表区 |
| **UI/交互规范** | 视觉样式，改了不影响功能 | 箭头图标、颜色、滚动条样式 |
| **代码实现逻辑规范** | 命名/逻辑/后端 | 命名规范、API 规范、分页逻辑 |

**核心判断方法**：改了之后影响功能吗？
- 影响 → 组件规范
- 不影响 → UI/交互规范

---

## 二、文档重组

根据新的分类标准，对现有文档进行了重组：

```
docs/
├── standards/
│   ├── component-standard.md       # 前端组件规范
│   ├── ui-interaction-standard.md  # UI/交互规范
│   └── logic-standard.md           # 代码实现逻辑规范
└── API_business_project.md        # 业务文档

.trae/rules/
└── project_rules.md                # 约束性规则
```

---

## 三、Skill 与文档的边界讨论

### 问题

用户担心 `project_rules.md` 堆积太多内容会增加阅读负担。

### 讨论过程

用户指出：
- Git 上传应该放 Skill，不是规则（因为需要用户显式指令触发）
- 组件规范、代码规范应该放文档

### 最终结论

| 类型 | 判断标准 | 存放位置 |
|------|----------|----------|
| **Skill** | 用户显式指令触发、有明确操作步骤 | `.trae/skills/<name>/SKILL.md` |
| **文档** | 按需参考、不需要每次扫描 | `docs/` 目录下相应文档 |

**规则更新**：当用户说"我多次告诉你应该..."等类似表达时，AI 应意识到需要将内容沉淀到文档/规则/Skill中。当不确定记录到哪一类时，必须先与用户澄清。

---

## 四、Git 上传流程的讨论

### 问题

Git 上传流程应该放哪里？

### 结论

- **本质**：用户显式指令触发 + 有明确操作步骤
- **判断**：符合 Skill 的典型场景
- **结果**：创建 `git-upload` Skill，移除规则文件中的流程说明

### 额外讨论

- 用户说"我每次任务结束会给你说上传 GitHub"
- 所以 Git 上传是**用户指令触发**，不是自动执行
- 规则改为：**"任务结束"的判断标准**是用户明确说"任务结束"、"今天就到这里"等

---

## 五、任务结束规则的明确

### 问题

我之前误以为"执行完毕" = "任务结束"，导致不必要的 Git 上传。

### 讨论结果

规则明确：
- **"任务结束"的判断标准**：用户明确说"任务结束"、"今天就到这里"等
- **"任务结束"的例外**：仅更新规则文件、确认方案等操作**不算**任务结束，**必须询问**"需要上传到 GitHub 吗？"

---

## 六、讨论记录机制的建立

### 需求

用户希望记录每次对话讨论的内容，形成沉淀。

### 讨论过程

- 用户问：应该每次我提醒你总结，还是你判断什么时候开始总结？
- 用户补充：一天内可能有多个任务，怕我忘记询问

### 最终方案

**混合模式**：
- 当用户说"任务结束"或"今天就到这里"时，询问是否需要总结
- 在对话过程中，将值得总结的讨论记录到 todo list
- 讨论记录存放目录：`docs/discussions/`，文件命名格式：`YYYY-MM-DD-主题.md`

---

## 七、规则与 Skill 的内容边界

### 问题

用户发现规则文件中还有 Git 上传相关的内容，但之前不是说要把 Git 上传放到 Skill 吗？

### 讨论结果

**规则文件中保留规则，Skill 中保留执行流程**：

| 内容 | 位置 | 说明 |
|------|------|------|
| "每次任务结束时，必须将代码提交到 GitHub" | 规则文件 | **规则**：约束 AI 的行为 |
| 版本号递增规则 | 规则文件 | **规则**：约束 AI 的行为 |
| Git 上传的具体步骤（分开执行命令、验证等） | Skill | **执行流程**：如何操作 |

简单说：
- **规则文件** → 告诉 AI **什么时候要上传**
- **Skill** → 告诉 AI **怎么上传**

---

## 八、"讨论"与"执行"的区分

### 问题

我经常在用户说"你是怎么理解的呢"时，就直接开始执行操作，而不是先讨论澄清。

### 讨论结果

在"需求理解规则"中补充：
- **当用户说"你是怎么理解的"、"我理解对吗"等类似表达时**，是希望讨论而非执行，不要急于执行操作

---

## 九、规则文件压缩

### 问题

Trae 规则文件建议字符数不超过 1000，当前规则文件有 2336 字符。

### 讨论结果

将"强制执行的组件规范"移到 component-standard.md，规则文件只保留纯 AI 行为约束。

**压缩后规则文件结构：**
- 任务结束规则
- 需求理解规则
- 任务执行规则
- 文档更新规则
- 讨论记录规则

**压缩后字符数：697 字符**（原 2336 字符）

**最终规则文件内容：**
```markdown
# API Business 项目规范

## 约束性规则

### 任务结束规则
- 判断：用户说"任务结束"、"今天就到这里" | 例外：仅更新规则、确认方案 → 询问"需要上传GitHub吗？"
- 执行：提交GitHub并打tag，版本号末位满10进1，完成告知版本号

### 需求理解规则
- 未完全理解或有歧义时必须澄清，不假设意图
- 用户说"你是怎么理解的"时是讨论而非执行

### 任务执行规则
- 一次只做一件事，确认后再做下一件 | 不确定时先提方案

### 文档更新规则
- 确认决策后主动更新 | 用户说"多次告知应..."时需沉淀
- 不确定归属时先澄清
- 归属：约束AI行为→project_rules.md | 前端组件→component-standard.md | 视觉样式→ui-interaction-standard.md | 命名/逻辑→logic-standard.md | 业务知识→API_business_project.md
- 显式指令+操作步骤 → Skill (.trae/skills/<name>/SKILL.md)

### 讨论记录规则
- 用户说"任务结束"时询问是否总结，过程记录到todo list
- 存放：docs/discussions/YYYY-MM-DD-主题.md
```

---

## 总结

这次讨论解决了几个重要问题：

1. **文档分类标准**：建立了组件规范 vs UI/交互规范的判断标准
2. **Skill vs 文档边界**：明确了什么内容适合放 Skill，什么适合放文档
3. **任务结束规则**：明确了"任务结束"的判断标准和例外情况
4. **讨论记录机制**：建立了讨论沉淀的流程
5. **规则与 Skill 的边界**：明确了规则文件管"何时执行"，Skill 管"如何执行"
6. **讨论与执行的区分**：明确了讨论意图时不急于执行
7. **规则文件压缩**：将组件规范移出，压缩到 697 字符

这些讨论体现了"在实践中不断完善规则"的思想——每次遇到问题并解决后，把最优方案沉淀到规则中，确保后续不再犯同样的错误。
