# API商业化业务现状梳理

日期：2026-04-22

---

## 一、角色清单（7个）

| 角色 | 部门 | 核心职责 |
|------|------|----------|
| 渠道 | - | 拓展供应商、签合同 |
| 运营 | - | 核心角色：审批、上架、配置通道和策略 |
| 技术测试 | - | 执行测试、必要时技术处理 |
| 法务 | - | 审核供应商/客户合同（OA系统） |
| 销售 | - | 拓展客户、谈需求 |
| 财务 | - | 结算、催款、开票 |
| 客服 | - | 处理客户问题、转报技术排查 |

## 二、核心业务流程

### 供应商接入流程

1. 渠道和供应商谈合作 → 线下签合同
2. OA备案 → 法务审核
3. 人工在业务系统提交通道上线测试流程（OA与业务系统未打通）
4. 运营审批 → 技术测试
5. 如需技术处理则技术介入
6. 测试通过 → 通道上架 + 产品关联通道
7. 测试不合格 → 渠道反馈供应商，整改后重测或放弃

**不合格原因**：延迟太高、输出参数不满足要求、输出数据查得率太低等

### 客户购买流程

1. 销售拓展客户 → 谈需求 → 签合同
2. 客户注册账号 + 企业认证
3. 客户申请API → 法务审批（是否在合同范围内）
4. 审批通过 → 预付费门户下单 / 后付费运营后台配置
5. 运营根据价格/体量配置通道（及备用/切换策略）
6. 获取APIkey → 调用API

### 结算流程（后付费）

1. 和客户协商结算周期（每个客户周期可能不同）
2. 系统拉取调用明细并计算应付款
3. 财务和客户核对，双方一致后打款
4. 平台给客户开发票

## 三、客户分层

| 类型 | 付费方式 | 特点 |
|------|----------|------|
| 小客户 | 预付费为主 | 金额一般几万 |
| 大客户 | 后付费为主 | 按周期结算，信用额度管理 |

---

## 四、计费方式

| 方式 | 说明 |
|------|------|
| 按调用次数 | 每调一次多少钱 |
| 套餐包 | 固定价格买固定次数 |
| 阶梯计费（累进） | 每区间单独计算后累加，如0-100次0.2元，超过100次0.3元，则150次=0.2×100+0.3×50 |
| 阶梯计费（到达） | 超阈值后全部按该阶梯单价，如150次=0.3×150 |
| 按实体计费 | 按车牌/企业收固定费用 |

## 五、系统清单

| 系统 | 职责 | 备注 |
|------|------|------|
| API管理系统 | 核心：供应商/通道/产品/价格/计费/监控/转发请求 | 客户调用请求打到此系统 |
| 客户自助门户 | 下单、获取APIkey | 仅用于下单/获取密钥 |
| OA系统 | 合同备案审核 | 与业务系统未打通 |

### 系统关系

- 客户调用API时请求打到API管理系统，由系统转发到供应商
- 门户仅用于下单/获取密钥，实际调用不走门户

## 六、核心数据对象

| 数据对象 | 说明 |
|----------|------|
| 供应商 | 基本信息 |
| 通道 | 入参/出参、成本、价格、状态 |
| 产品 | 对应通道列表 |
| 客户 | 企业信息 |
| 账户 | 资金账户（余额、信用额度） |
| API密钥 | 调用凭证 |
| 订单 | 预付费/后付费 |
| 账单 | 结算周期、金额 |
| 调用记录 | 计费依据 |
| 价格策略 | 计费规则 |

---

## 七、产品与通道关系

- 一个产品可配置多条通道（入参/出参格式相同，可互换）
- 运营根据客户购买价格、体量、通道成本/价格/质量决定配置哪条通道
- 重要客户会配置备用通道和自动切换策略（如失败2次自动切换）
- 产品上架时配置通道，但针对每个客户的通道交付需运营单独配置

## 八、待补充/待优化

| 事项 | 说明 |
|------|------|
| 信用额度机制 | 计划引入，通过系统限制替代人工操作，避免坏账 |
| 通道质量指标 | 运营人工判断，参考并发、成功率等 |
| 自动续费 | 预付费可设置，次数用完前自动扣款续费，系统不主动提醒 |
