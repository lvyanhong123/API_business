### 2. 账号体系概念澄清

**重要澄清**：
- `customers` 表 = 企业客户主数据
- `customerAccounts` 表 = 企业资金账户（余额、信用额度）
- **新概念：账号 (accounts)** = 员工/个人登录账号
- 一个企业可以有多个账号
- 一个账号可以绑定多个企业
- 账号可绑定企业也可不绑定（纯个人账号）

**账号与企业的关系**：
```
账号（员工/个人）
    ├── 未绑定企业 → 使用个人账号
    └── 绑定到企业 → 使用企业的资金账户
```

### 3. 企业管理员机制

- 创建企业的账号自动成为该企业的管理员
- 企业申请更换管理员 → 平台管理员审批
- 平台管理员可在后台直接创建企业并指定管理员

### 4. API Keys 分类

| 类型 | 关联 | 说明 |
|------|------|------|
| 企业API Key | customerId | 登录企业状态可用 |
| 个人API Key | accountId | 仅个人账号使用 |

## 决策与结论

### 功能完成标准

**完整功能 = 后端接口 + 前端页面 + 前后端联调通过。只做后端接口不算完成。**

这条应纳入 logic-standard.md（开发实现规范），不是 project_rules.md（约束AI行为）。

### 多用户端考虑

实现任何功能时，必须同时考虑所有涉及的用户端：
- 客户自助门户 (client.html, 端口3001)
- 管理后台 (admin.html, 端口3000)

### 功能完成清单

1. **账号体系后端** - accounts、accountCustomers 表 + API
2. **客户门户前端** - client.html 改造（注册/登录/企业切换/绑定）
3. **管理后台账号管理** - 账号列表、创建账号、绑定企业、审批
4. **管理后台创建客户** - 管理员可创建企业客户
5. **账单管理菜单** - 新建菜单组整合调用明细/客户账单/供应商账单

### 历史遗留问题处理

删除了不适用的测试文件（mongoose模型测试，与项目实际架构不匹配）。
