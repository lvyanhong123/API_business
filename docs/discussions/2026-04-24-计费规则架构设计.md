# 讨论总结：API商业化平台计费规则架构设计

**日期**：2026-04-24

---

## 讨论背景

用户提出API商业化平台当前计费规则难以拓展，需要分析原因并设计可扩展的计费规则架构。

---

## 问题分析

### 当前系统问题
- 当前支持：按次计费、部分阶梯、部分四网、套餐包
- 当前不支持：多接口赠送、多接口合计阶梯、多接口最大计费、计费键、手工记录

### 根因
1. 调用明细表缺少通用计量字段（billing_key、operator）
2. 计费规则缺少"计费维度"抽象
3. 缺少接口分组概念
4. 计费配置结构单一

---

## 设计方案

### 四层架构
```
调用明细层 → 规则模板层 → 接口组层 → 客户定价层 → 计算引擎层
```

### 规则模板五要素
```
规则模板 = 计费维度 + 计费键类型 + 组合策略 + 定价模型 + 参数配置
```

1. **计费维度**：谁来计量（SINGLE_API / API_GROUP / BILLING_KEY / OPERATOR / MANUAL）
2. **计费键类型**：键是什么（CAR / ENTERPRISE / LICENSE / DEVICE / CUSTOM）
3. **组合策略**：多接口怎么组合（仅API_GROUP时：SUM / MAX / PRIMARY_BONUS）
4. **定价模型**：怎么算（FIXED / TIERED / BRANCH / CAPPED / PACKAGE）
5. **参数配置**：具体填什么价格（支持单值、有序列表、key-value映射、嵌套对象）

---

## 表结构改造

### 新建表
- **规则模板表**：存储五要素定义
- **接口组表 + 成员表**：定义API组合关系

### 改造表
- **调用明细表**：新增计费键、计费键类型、运营商、原始数据字段
- **客户定价表**：支持绑定API或接口组，引用规则模板

---

## 改造优先级

| 优先级 | 内容 | 目标 |
|--------|------|------|
| P0 | 接口组表 + 客户定价表扩展 | 支持多接口赠送、合计阶梯、最大计费 |
| P1 | 规则模板表 | 为新规则提供模板支撑 |
| P2 | 调用明细表扩展 | 支撑计费键计费 |
| P3 | 计算引擎优化 | 长期目标，减少新规则开发量 |
