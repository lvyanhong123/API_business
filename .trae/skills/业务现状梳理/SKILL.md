# B端业务现状梳理

基于TOGAF（简化）、BABOK、RACI框架，结构化梳理B端业务现状。

## 使用场景

- 新项目启动前的业务调研
- 需求分析前的现状了解
- 系统重构前的业务梳理
- 跨部门业务对齐

## 输入模式

本Skill支持**两种输入模式叠加**：

### 模式1：结构化访谈记录（客观信息）

当输入包含以下特征时，自动识别为访谈记录模式：

- 包含"访谈对象"字段
- 包含"问题X："格式的问题
- 包含"回答："格式的记录

**处理方式**：
1. 解析结构化字段，提取角色、问题、回答
2. 按角色/主题聚合信息
3. 识别关键事实、痛点、流程描述

### 模式2：对话输入（主观分析）

当输入为对话式描述时：

- 用户主动提供的分析性信息
- 助理基于观察的补充说明
- 基于访谈记录的个人理解

**处理方式**：
1. 解析对话内容，提取关键信息
2. 与已有信息交叉验证
3. 补充客观信息中可能缺失的分析视角

### 叠加模式（默认）

**两种模式可同时存在，自动叠加整合：**

```
结构化访谈记录（客观）
    +
对话输入（主观+分析）
    ↓
整合 → 更全面的业务现状
```

**整合逻辑**：
1. 客观信息（访谈记录）提供事实依据
2. 主观信息（对话）提供分析视角
3. 两者矛盾时，标记为"待验证"并主动询问
4. 两者一致时，互相印证增强可信度

## 交互方式：渐进式访谈

本skill采用**多轮小步访谈**而非一次性问卷。

### 核心原则

1. **一次只问一个问题**：每个问题独立回答，减少认知负担
2. **根据回答推进**：根据用户反馈动态调整下一个问题
3. **实时记录摘要**：每轮回答后简要复述关键点，让用户确认理解正确
4. **矛盾时二次确认**：发现逻辑不一致时，主动追问澄清
5. **阶段性汇总**：每完成一个阶段，生成阶段性汇总供用户确认
6. **等待确认后才推进**：每次提问后必须等待用户回复，不在用户未确认前自行提出下一个问题
7. **注意访谈对象背景**：如果对方不是技术出身，避免问纯技术实现问题；如需了解技术细节，应找技术同学而非产品经理
8. **优先业务视角**：从角色、流程、责任角度提问，而非技术实现角度
9. **主动判断遗漏**：由我来判断还有哪些遗漏并主动深挖，不把这个问题抛给访谈对象

### 访谈状态机

```
边界确认 → 组织架构 → 业务流程 → 系统清单 → 系统集成 → 数据实体 → 汇总确认
```

每轮只做一件事：
- 提问
- **等待用户回复/确认**
- 摘要确认（"我理解你说是XXX，对吗？"）
- 记录 → 下一个问题

如果用户没有直接确认，而是补充了信息或者纠正了理解，则：
- 根据新的信息更新理解
- 再次确认
- 确认无误后再推进

### 摘要格式（每轮回答后生成）

每轮访谈结束后，我会生成如下摘要供你确认：

```
【已确认】
- 角色A：负责XXX
- 角色B：负责XXX

【待确认】
- 角色C：？（需要进一步了解）
```

### 进入下一阶段的条件

每个阶段只有在用户确认阶段性汇总后，才进入下一个阶段。不强行推进。

## 阶段一：业务架构梳理

### 1.1 组织结构

本阶段目标是了解内部角色构成和核心职责。

**访谈入口**：直接请用户描述涉及哪些部门/角色，不需要提前准备任何材料。

**访谈问题示例**：

第一轮：
> "请介绍一下目前涉及API商业化业务的有哪些部门/角色？"

第二轮（追问细节）：
> "【角色A】这个岗位在API业务中的主要职责是什么？"

**输出模板**：

| 角色 | 所属部门 | 核心职责 | 备注 |
|------|----------|----------|------|
| - | - | - | - |

**摘要确认**：每确认一个角色，复述一次，用户确认后记录。

### 1.2 角色职责（RACI）

本阶段目标是明确各角色在业务流中的责任归属。

**访谈时机**：在组织架构确认完毕后进行。

**RACI说明**：
- **R (Responsible)**：负责执行
- **A (Accountable)**：最终批准
- **C (Consulted)**：双向沟通
- **I (Informed)**：单向通知

**访谈问题示例**：

第一轮（针对具体流程）：
> "以【供应商接入API】这个流程为例，哪些角色会参与？各自负责什么环节？"

第二轮（补充其他流程）：
> "那【客户购买API】这个流程呢？有哪些角色参与？"

**输出模板**：

| 业务场景 | 角色R | 角色A | 角色C | 角色I |
|---------------|-------|-------|-------|-------|
| 供应商接入 | - | - | - | - |
| 客户购买 | - | - | - | - |

**摘要确认**：每确认一个流程，复述一次关键分工，用户确认后记录。

### 1.3 业务流程

本阶段目标是梳理核心业务的完整流程和步骤。

**访谈时机**：在角色职责确认后进行，或并行进行（取决于用户对流程的熟悉程度）。

**访谈问题示例**：

第一轮（主流程）：
> "请描述一下API从供应商接入到最终给客户使用的完整流程，越详细越好，包括每个环节的动作和参与角色。"

第二轮（异常/分支）：
> "在这个流程中，有没有一些特殊的情况需要处理？比如供应商API不合格怎么办？客户欠费了怎么办？"

**输出模板**：

- **业务流程泳道图**：按角色分泳道

| 步骤 | 泳道（角色） | 动作 | 输入 | 输出 | 备注 |
|------|-------------|------|------|------|------|
| 1 | - | - | - | - | - |

**摘要确认**：先确认整体流程框架，再逐个补充细节。

---

## 阶段二：应用架构梳理

### 2.1 系统清单

本阶段目标是了解现有的系统有哪些，以及各自的职责边界。

**访谈时机**：可以在业务流程确认后进行（因为系统是支撑流程的）。

**访谈问题示例**：

第一轮：
> "目前API商业化业务主要用到了哪些系统？"

第二轮（追问细节）：
> "【系统A】这个系统主要管什么？有哪些主要功能模块？"

第三轮（负责人）：
> "【系统A】的日常运维和支持是谁负责的？"

**输出模板**：

| 系统名称 | 核心功能 | 归属（自研/外采） | 负责人 | 备注 |
|----------|----------|------------------|--------|------|
| - | - | - | - | - |

**摘要确认**：每确认一个系统，复述其核心定位，用户确认后记录。

---

### 2.2 系统集成关系

本阶段目标是了解系统之间的调用关系和数据流向。

**访谈时机**：在系统清单确认后进行。

**访谈问题示例**：

第一轮：
> "这些系统之间是怎么配合的？比如客户购买API这个过程，数据在哪些系统之间流转？"

第二轮（追问）：
> "客户调用API的时候，请求是先到哪个系统？再到哪里？"

**输出模板**：

- **系统集成关系图**：用箭头表示数据流向

| 源系统 | 目标系统 | 接口/数据内容 | 调用方式 | 备注 |
|--------|----------|---------------|----------|------|
| - | - | - | - | - |

**摘要确认**：画出简化的系统关系图，让用户确认是否正确。

---

## 阶段三：关键数据实体

### 3.1 核心业务对象

本阶段目标是识别系统中的核心业务数据。

**访谈时机**：在系统集成关系确认后进行，或并行进行。

**访谈问题示例**：

第一轮：
> "在这个业务里，最核心的数据对象有哪些？比如供应商信息、客户信息、API产品信息、订单信息、账单信息？"

第二轮（追问）：
> "【数据对象A】里面最关键的字段是什么？"

**输出模板**：

| 数据实体 | 所属系统 | 核心字段 | 说明 |
|----------|----------|----------|------|
| - | - | - | - |

---

### 3.2 数据流转

本阶段目标是了解核心数据在各系统间的流转路径。

**访谈时机**：在核心业务对象确认后进行。

**访谈问题示例**：

第一轮：
> "以【客户下单购买API】为例，这个过程涉及哪些数据实体？数据是怎么在系统间传递的？"

**输出模板**：

- **数据流转图**：简化的数据流向图

---

## 输出汇总

每完成一个阶段，生成阶段性汇总；全部完成后，汇总全量文档。

### 阶段性汇总（每阶段结束时生成）

```
【阶段一：业务架构 - 已确认】
- 角色清单：渠道、销售、运营...
- 核心流程：供应商接入 → API上架 → 客户购买 → 计费结算

【阶段一：待确认】
- 异常流程分支（需要补充）

【阶段二：应用架构 - 已确认】
- 系统清单：API管理平台、客户门户...
- 系统关系：...

【阶段三：关键数据 - 已确认】
- ...
```

### 最终输出

1. **业务架构图**：组织结构 + 角色 + 流程
2. **应用架构图**：系统清单 + 集成关系
3. **数据流转图**：核心数据 + 流转路径

## 信息来源标记

在整合信息时，明确标注信息来源：

| 标记 | 含义 | 可靠度 |
|------|------|--------|
| 【访谈-角色A】 | 来自结构化访谈记录的客观回答 | 高 |
| 【助理分析】 | 来自助理的主观分析和补充 | 中 |
| 【待验证】 | 访谈与对话存在矛盾，需进一步确认 | 待定 |

## 注意事项

1. **保持客观**：现状梳理是描述"现在是什么样"，不评判好坏
2. **边界清晰**：开始前明确梳理范围，避免发散
3. **验证确认**：每个阶段产出需与业务方对齐后再继续
4. **不含需求**：痛点、改进点、优先级等需求分析内容不在本skill范围内
5. **注意访谈对象背景**：如果对方不是技术出身，避免问纯技术实现问题；如需了解技术细节，应找技术同学而非产品经理
6. **优先业务视角**：从角色、流程、责任角度提问，而非技术实现角度
7. **主动判断遗漏**：由我来判断还有哪些遗漏并主动深挖，不把这个问题抛给访谈对象
8. **双模式整合**：结构化访谈记录提供事实，对话输入提供分析，两者互补，矛盾时主动标记并询问
