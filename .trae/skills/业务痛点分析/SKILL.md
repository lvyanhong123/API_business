# 业务痛点分析

基于业务现状梳理结果，结合补充访谈，系统性识别、分析和定级业务痛点。

## 核心定位

| 维度 | 描述 |
|------|------|
| 输入 | 业务现状梳理报告、补充访谈记录 |
| 输出 | 痛点清单、根因分析、优先级排序、改进机会 |
| 目的 | 发现改进机会，为需求拆解提供依据 |

## 整合方法论

| 阶段 | 融入方法论 |
|------|-----------|
| 痛点识别 | JTBD（Jobs-to-be-Done）+ 用户旅程图 |
| 痛点验证 | 访谈追问 + 量化确认 |
| 根因追溯 | 5 Why + 鱼骨图（人机料法环测）|
| 痛点定级 | 影响范围/频率 + Kano模型 |
| 优先级排序 | Impact vs Effort + 业务战略对齐 |

## 基础原则

1. **分析师自主判断**：痛点优先级由分析师根据线索可信度、影响范围、商业价值自主判断，无需询问用户。判断后直接说明理由，让用户确认。
2. **不假设**：不预设"这肯定是痛点"，一切从访谈和现状中推导
3. **区分层次**：现象痛点 vs 结构痛点，避免头痛医头
4. **量化优先**：无法量化的痛点至少要定性准确
5. **关联业务目标**：痛点必须连接到业务价值，否则优先级存疑

## 输入要求

开始痛点分析前，确认已有以下输入之一：
- 业务现状梳理报告（完整或阶段性）
- 或：明确本次分析的边界范围

## 分析框架

```
痛点识别 → 痛点验证 → 根因追溯 → 定级评估 → 优先级排序 → 输出汇总
```

### 第一阶段：痛点识别

#### 1.1 从现状梳理中提取线索

基于现状梳理报告，标注以下潜在问题区域：
- 流程中断或绕路的地方
- 多角色协作的交接地带
- 手动操作密集的环节
- 异常处理逻辑不清晰的地方
- 依赖外部系统的瓶颈点

**标注格式**：

| 线索 | 来源（流程/系统/角色） | 可信度 | 待验证 |
|------|----------------------|--------|--------|
| - | - | 高/中/低 | 是/否 |

#### 1.2 补充访谈设计

针对潜在痛点线索，设计专项访谈问题。

**访谈入口**：请用户描述"不顺"和"担心"的场景。

**访谈问题类型**：

| 类型 | 问题示例 |
|------|----------|
| 效率类 | "在这个流程里，哪个环节最费时间？" |
| 质量类 | "上线的时候最怕出什么问题？" |
| 协作类 | "跟XX部门协作的时候，最大的分歧是什么？" |
| 稳定性类 | "系统最容易在哪个环节出问题？" |
| 成本类 | "这块业务最费人力的地方在哪？" |

**追问原则**：
- "能举个具体例子吗？"（避免抽象回答）
- "上一次出问题是什么时候？"（验证真实性）
- "当时怎么处理了？"（了解当前解法）

#### 1.3 JTBD框架应用

对已识别的痛点，用Jobs-to-be-Done框架追溯用户真实动机。

**JTBD句式**：
```
When（情境）→ 我在XXX情境下
I want to（动机）→ 我想要XXX
So I can（目标）→ 以便我能达成XXX
```

**示例**：

| 痛点 | When（情境） | I want to（动机） | So I can（目标）|
|------|-------------|------------------|-----------------|
| 审批流程长 | 想让API快速上线 | 减少反复沟通确认 | 早点让客户用上 |
| 不知道API效果 | 想优化API定价 | 了解调用数据 | 提升收益 |

**访谈追问**：
- "你当时想解决什么问题？"
- "如果不解决这个问题，会怎样？"
- "你觉得最好的解决方式是什么样的？"

#### 1.4 用户旅程图构建

基于现状梳理的业务流程，构建跨角色用户旅程图。

**模板**：

| 阶段 | 触点 | 用户行为 | 用户想法 | 情绪（高↑/低↓）| 痛点 |
|------|------|----------|----------|----------------|------|
| 接入申请 | - | - | - | - | - |
| 资质审核 | - | - | - | - | - |
| 技术对接 | - | - | - | - | - |
| 测试验收 | - | - | - | - | - |
| 正式上线 | - | - | - | - | - |

**绘制步骤**：
1. 从现状梳理获取流程步骤
2. 识别每个步骤的触点和角色
3. 访谈获取"想法"和"情绪"
4. 情绪低谷处标注为痛点候选
5. 追问低谷原因，验证痛点真实性

### 第二阶段：痛点验证

#### 2.1 痛点确认

将识别出的痛点与访谈对象确认，排除误判。

**确认格式**：

```
【痛点A】
- 描述：...
- 发生场景：...
- 影响角色：角色X、角色Y
- 发生频率：频繁/偶尔/罕见
- 验证状态：已确认/待澄清/已排除
```

#### 2.2 根因追溯

对已确认的痛点，连续追问5Why，找到根本原因。

**5Why示例**：

```
痛点：供应商API接入审批慢
Why1：审批流程长（3个角色共5个环节）
Why2：每个角色都要单独确认
Why3：没有并行处理机制
Why4：系统不支持多节点同时处理
Why5：（结构原因）API接入模块设计时未考虑并行审批
                    ↓
根因：系统功能缺失，而非流程设计问题
```

**鱼骨图（Ishikawa）根因归类**：

| 维度 | 说明 | 示例 |
|------|------|------|
| **人** | 角色能力、协作意愿 | 审批人经验不足、不清楚标准 |
| **机** | 系统功能、性能 | 系统不支持并行操作 |
| **料** | 数据质量、信息完整性 | 供应商资质材料不全 |
| **法** | 流程设计、审批规则 | 审批节点过多、无标准流程 |
| **环** | 组织架构、激励机制 | 部门KPI冲突、责任不清 |
| **测** | 监控告警、验收标准 | 缺乏验收标准、问题发现晚 |

**输出格式**：

| 痛点 | 根因层级 | 鱼骨维度 | 根因类型 | 结构/现象 |
|------|----------|----------|----------|-----------|
| - | 表层/中层/深层 | - | 功能缺失/流程低效/组织障碍/系统限制 | - |

### 第三阶段：痛点定级

#### 3.1 评估维度

| 维度 | 评估标准 | 分值 |
|------|----------|------|
| **影响范围** | 多少人/角色受影响 | 1-5 |
| **发生频率** | 多久发生一次 | 1-5 |
| **持续时间** | 每次持续多久 | 1-5 |
| **损失程度** | 对业务目标的直接影响 | 1-5 |
| **紧急程度** | 积压多久了/是否恶化 | 1-5 |

#### 3.2 Kano模型分类

将痛点/需求分为五类，辅助判断价值方向：

| 类型 | 说明 | 痛点示例 |
|------|------|----------|
| **基本型** | 必须有，否则用户极度不满 | 支付失败、数据不安全 |
| **期望型 | 越好用越满意，越差越不满 | 审批速度、界面体验 |
| **兴奋型** | 没想到但很惊喜 | 自动推荐、智能提醒 |
| **无差异型** | 有没有无所谓 | 皮肤主题 |
| **反向型** | 做了反而用户不喜欢 | 强制绑定手机号 |

**Kano访谈问题**：
- "如果XXX功能实现了，你的感受是？"（兴奋）
- "如果XXX功能没有了，你的感受是？"（基本型）
- "你觉得XXX功能重要吗？"（判断期望型）

#### 3.3 优先级矩阵

```
                    影响范围
              小          大
         ┌─────────┬─────────┐
       高 │  重要   │  紧急   │
  紧     │         │         │
  急     ├─────────┼─────────┤
  度     │  观察   │  战略   │
       低 │         │         │
         └─────────┴─────────┘
```

**行动原则**：
- **紧急+重要**：立即处理
- **重要+影响大**：规划迭代
- **紧急+影响小**：快速修复
- **观察项**：持续监控，暂不投入

**综合排序参考**：
1. 基本型痛点 > 期望型痛点 > 兴奋型痛点
2. 紧急程度高 > 低
3. 影响范围大 > 小

### 第四阶段：输出汇总

#### 4.1 痛点清单

| ID | 痛点描述 | 影响角色 | 根因 | Kano类型 | 影响范围 | 发生频率 | 优先级 |
|----|----------|----------|------|----------|----------|----------|--------|
| P1 | - | - | - | - | - | - | 高/中/低 |

#### 4.2 改进机会

| 痛点ID | 可能的改进方向 | 预计价值 | 备注 |
|--------|---------------|----------|------|
| P1 | - | 高/中/低 | - |

**注意**：改进方向是方向而非具体方案，方案由后续需求拆解和产品设计产出。

#### 4.3 阶段性汇总格式

```
【痛点分析 - 已确认】
- 痛点清单：P1、P2、P3
- 高优先级痛点：P1（基本型，影响范围广、频发）
- 根因分析：P1根因是系统功能缺失（鱼骨：机）

【待确认】
- P4痛点需要再次访谈确认发生频率
- P5需要技术侧评估系统改造难度
```

## 交互方式

与业务现状梳理类似，采用**渐进式多轮访谈**：
1. 每轮只问一个痛点维度
2. 确认后再推进
3. 每轮生成摘要让用户确认
4. 阶段性汇总后等待确认再继续

## 与现状梳理的衔接

| 现状梳理输出 | 痛点分析输入 |
|--------------|--------------|
| 角色清单 | 受影响角色映射 + 用户旅程图触点 |
| 业务流程 | 痛点场景定位 + 情绪低谷识别 |
| 系统清单 | 问题归因（鱼骨图：机）|
| 数据流转 | 痛点数据影响分析 |
| 组织架构 | 鱼骨图：环、法 |

## 注意事项

1. **不评判现状**：痛点是改进机会，不是追责依据
2. **区分需求和痛点**：痛点分析产出是"问题清单"，不是"功能清单"
3. **优先级是判断，不是计算**：矩阵是参考，最终决策需结合业务战略
4. **避免引导**：访谈时不问"你觉得XXX是不是问题"，而问"你觉得哪里不顺"
5. **JTBD追问深度**：不止问"哪里不顺"，还要问"不顺的时候你想达成什么"
